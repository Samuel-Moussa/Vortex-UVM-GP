# VECADD Kernel Runtime Analysis

## 1. Executive Summary

`vecadd` is functionally correct, but it is much slower than `hello` because it does substantially more work at startup and during kernel launch.

The final successful run completed after `3817046` cycles and produced the expected vector-add output. The apparent "hang" is really a long runtime path caused by:

- startup `memset` of the static heap / `.bss`
- heavy host-side `vx_printf` traffic
- thread-launch and per-thread runtime overhead from `vx_spawn_threads`
- kernel-side output from each thread

At the time of the long-running trace, `vecadd` had a 1 MB static pool in `.bss`:

- `#define HEAP_SZ 1024 * 1024`
- `char __data_pool[HEAP_SZ];`

That large BSS footprint increases startup zeroing time significantly. Even when the test is correct, the simulation can take millions of cycles before termination.

## 2. Symptoms Observed

### 2.1 Testbench behavior

- `vif.status_if.busy` remains high for long durations.
- `per_cluster_busy=1` remains high.
- `ebreak_detected=0` remains low during the long startup/runtime phase.
- The test eventually completes and watchdog disarms once termination is reached.

### 2.2 Probe behavior

Pipeline probe output repeatedly shows:

- `PC=0x80000948` with alternating instructions like `0x00a282b3` and `0xfc0e88e3`
- periodic visits to `PC=0x80003ae4` and `PC=0x80003af8`
- `ebreak_seen=0`
- `exit_addr_seen=0`

This indicates the program is still in runtime/control loops and has not yet reached the end of `main()` or the `_Exit` path.

### 2.3 Runtime comparison with `hello`

The difference versus `hello` is mostly workload, not correctness:

- `vecadd` dump contains about `74` `vx_printf` call sites and `42` `vx_spawn_threads`-related call sites.
- `hello` dump contains only `2` `vx_printf` call sites and no `vx_spawn_threads` path.
- `vecadd` prints the full source buffers, the destination buffer, kernel launch diagnostics, and one message per spawned thread.

That output volume alone makes `vecadd` much slower in simulation, even before accounting for startup BSS zeroing.

## 3. Forensic Evidence

## 3.1 Disassembly: startup flow reaches runtime init before main

From `vecadd.dump` startup:

- `_start` executes runtime setup
- `_start` calls `memset`
- `_start` then calls `main`

Sequence includes:

- `0x80000060: jal ... <memset>`
- `0x80000068: jal ... <main>`
- `0x8000006c: j <exit>`

This is normal C runtime behavior: zero BSS before entering `main`.

## 3.2 Disassembly: loop addresses map to memset loop body

Addresses seen in probes match `memset` loop instructions:

- `0x80003ae4: sw a2, 0x8(t1)`
- `0x80003af8: sw a2, 0x1c(t1)`
- nearby loop control branch at `0x80003b0c: beqz ... <memset+0x120>`

Therefore, repeated execution at these addresses is expected while clearing large memory ranges.

## 3.3 Symbol and section sizes confirm large BSS workload

`llvm-size -A vecadd.elf` reports:

- `.bss` size: `1048600` bytes

`llvm-nm` confirms:

- `__data_pool` in BSS region
- `__data_pool` start and BSS end span roughly 1 MB

This directly matches the static pool declaration in `vecadd.cpp`.

## 3.4 End-of-program control flow

The end of `main()` is reached at:

- `0x800008ac: li a0, 0x0`
- `0x800008e0 <_Exit>` begins immediately afterward
- `0x800008e4: jal vx_perf_dump`
- `0x800008e8: sw a0, 0x0(t0)` with `t0 = 0x88`
- `0x800008f0: vx_tmc zero`

So the completion path is the normal `_Exit` sequence, not a missing-`ebreak` failure.

## 4. Why `hello` Works but `vecadd` Takes Much Longer

`hello` has a much smaller startup memory footprint and far less runtime printing, so it finishes quickly.

`vecadd` combines a large static BSS, many formatted prints, and thread-launch overhead. That combination pushes total runtime into the millions of cycles even though the result is correct.

## 5. Clarification on Program Termination Mechanism

Vortex normal termination path is not necessarily `ebreak`.

Observed runtime exit path:

- `_start` -> `main` -> `exit` -> `_Exit`
- `_Exit` performs:
  - `vx_perf_dump`
  - status write via MMIO (`sw a0, 0x0(t0)` where `t0=0x88`)
  - `fence`
  - `vx_tmc zero`

So a correct run can terminate via `_Exit` path without relying on `ebreak` detection.

## 6. Root Cause Statement

Primary causes of the long runtime:

- Large `.bss` generated by static `__data_pool[1024*1024]`.
- Heavy host/kernel logging with `vx_printf`.
- Thread spawn and per-thread runtime overhead from `vx_spawn_threads`.

Secondary contributing factor:

- The testbench watchdog initially interpreted the long runtime as a hang before the run later completed successfully.

## 7. Recommended Fixes

## 7.1 Immediate debug fix (fastest)

Reduce pool size for simulation debug builds:

- change `HEAP_SZ` from `1024 * 1024` to `4096` or `16384`

This should drastically reduce BSS clear time and allow reaching `main` completion sooner.

## 7.2 More robust medium-term fix

Move the large pool out of zero-init `.bss` path for simulation:

- place in a `noinit` section if linker script/toolchain supports it
- initialize explicitly in software only when needed

## 7.3 Testbench robustness

Add phased progress checks to distinguish:

- startup init progress
- kernel execution progress
- termination progress

and avoid treating long startup memset as deadlock.

## 8. Validation Plan

1. Rebuild `vecadd` with reduced `HEAP_SZ`.
2. Rerun `kernel_launch_test` with existing probes.
3. Confirm probe transitions:
  - leaves `memset` loop PCs (`0x80003ae4`, `0x80003af8`)
  - reaches `main` body and completion path
  - reaches `_Exit` and the exit MMIO write path (`0x88`)
4. Confirm busy deasserts and the test completes successfully, even if total cycles remain high.

## 9. Final Conclusion

This is not a functional failure.

The evidence shows `vecadd` is correct but expensive to simulate because of startup BSS zeroing, thread-launch overhead, and unusually heavy print traffic. That is why it takes `3817046` cycles versus roughly `7020` for `hello`.
